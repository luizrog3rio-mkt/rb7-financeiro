## ATENÇÃO — vereditos superados por decisões posteriores do Luiz

As decisões fechadas da unificação VALEM SOBRE este mapa. Onde o texto abaixo conflitar, prevalece:

1. **O app sobrevivente é o categorizador-fatura** (projeto Supabase `qdnqghefwjpeiidjlzjy`), NÃO o rb7-financeiro. A direção da migração se INVERTE em relação ao mapa: o código TS do rb7 é portado para cá e adaptado ao schema vivo; os dados reais (~519 transações Sicoob) NÃO migram de projeto. Toda recomendação de "migrar dados de B para A" / "converge para o projeto qnjjipkqjcmsmumcqltg" está SUPERADA.
2. **Schema final é EN, não PT.** As tabelas vivas (profiles, categories, auto_rules, invoices+account_id, transactions, purchase_items, purchase_item_categories) ficam intactas; tabelas novas da Fase 1c já nasceram em EN: companies, accounts, entries, bank_transactions, hotmart_sales (enums payable/receivable, pending/paid/overdue/cancelled, checking/credit_card/inter_company). Os nomes PT propostos no mapa (perfis, faturas, lancamentos, transacoes_ofx, regras_categorizacao, itens_compra/compras_anotadas, categorias_compras, vendas_hotmart) estão SUPERADOS — servem só como referência conceitual do desenho.
3. **O seed do rb7 era FICTÍCIO**: a empresa "Berta" e as contas "Cartão Sicredi RB7", "Cartão C6 RB7", "Cartão Sicredi Berta" NÃO existem. Qualquer de-para do mapa apontando para essas contas/empresas é inválido. Contas reais (ex.: cartão Sicoob da Lívia) são cadastradas em `accounts`.
4. **Env vars**: vale `VITE_SUPABASE_PUBLISHABLE_KEY` (padrão do categorizador), não `VITE_SUPABASE_ANON_KEY` como o mapa recomenda.
5. **RLS modelo de equipe `to authenticated using(true)`** — confirmado (Fase 1b), este veredicto do mapa bate com a decisão fechada.

## 1. Parser OFX

- **Veredicto**: semelhante
- **Lado A (rb7-financeiro)**: Parser dedicado src/lib/ofx.ts (extrato bancário Sicredi/C6, SGML e XML). split por <STMTTRN> sem exigir tag de fechamento (ofx.ts:26-28); extrairTag com regex `<TAG>([^<\r\n]*)` cobre SGML e XML (ofx.ts:10-15); FITID obrigatório (ofx.ts:35); data normalizada para ISO YYYY-MM-DD (ofx.ts:17-22); SINAL do valor preservado (créditos e débitos, ofx.ts:31-32); MEMO com fallback NAME (ofx.ts:33); retorna tipo TRNTYPE. Não decodifica entidades HTML. Dedupe fica no banco: unique(conta_id,fitid) (00001_schema.sql:94) + upsert ignoreDuplicates (ImportarOfx.tsx:63-66).
- **Lado B (categorizador-fatura)**: parseOFX inline em App.jsx:119-137 (fatura de cartão). Regex /<STMTTRN>([\s\S]*?)<\/STMTTRN>/g EXIGE tag de fechamento (App.jsx:121) — falha em OFX SGML puro sem </STMTTRN>; descarta TRNTYPE=CREDIT com valor>0 (pagamentos/estornos, App.jsx:132); valor vira Math.abs (App.jsx:134) — sinal perdido; data vira string 'DD/MM/YYYY' (App.jsx:130), não date SQL; decodifica &amp; (App.jsx:127); FITID opcional e nunca usado para dedupe; auto-categoriza no ato do parse (App.jsx:133). Sem fallback NAME, exige MEMO (App.jsx:131).
- **Unificação**: Unificável num parser só: manter src/lib/ofx.ts do A como base (mais robusto: SGML sem fechamento, FITID obrigatório, sinal preservado, data ISO, fallback NAME). Portar do B apenas o decode de entidades (&amp;→&). A regra 'fatura de cartão' (descartar/inverter créditos, abs) NÃO entra no parser: vira transformação pós-parse na camada de import, parametrizada pelo tipo da conta destino (corrente vs cartao_credito). Descartar parseOFX do App.jsx e a perda de sinal/CREDIT — armazenar sinal real e filtrar créditos só na visão de fatura. Auto-categorização sai do parser e vira passo separado do pipeline de import.

## 12. Modelo de tenancy / RLS (decisão central do merge)

- **Veredicto**: conflito
- **Lado A (rb7-financeiro)**: Compartilhado intra-equipe: RLS habilitado nas 7 tabelas mas políticas FOR ALL TO authenticated USING(true) WITH CHECK(true) em todas as tabelas de negócio (00001_schema.sql:162-167); NENHUMA tabela de negócio tem user_id; único vínculo a usuário é lancamentos.criado_por (auditoria, 00001:73) e perfis. Hardening: papel imutável via privilégio de coluna (20260609180002:22-23), update de perfil só próprio (20260609180002:27-31).
- **Lado B (categorizador-fatura)**: Isolamento por usuário: toda tabela tem user_id NOT NULL FK auth.users e policy FOR ALL USING (auth.uid()=user_id) sem WITH CHECK explícito (migration.sql:52-54,67-69,83-85,103-105,119-121,136-138). O client confia no RLS: quase nenhuma query filtra user_id explicitamente (única exceção App.jsx:788).
